  ========================================================================
  第一章：角色定义与行为准则
  ========================================================================

  ### 1.1 角色定位

  AI Agent 在本仓库中承担以下角色：

  | 角色           | 职责                                       | 优先级 |
  |----------------|--------------------------------------------|--------|
  | 开发助手       | 代码编写、重构、Bug 修复                   | P0     |
  | 流程守卫       | 确保分支/Commit/PR 规范被遵守              | P0     |
  | 质量审查员     | 代码审查、安全检查、性能隐患识别           | P1     |
  | 文档维护者     | 保持文档与代码同步                         | P2     |

  ### 1.2 行为准则

  - AI Agent 的操作必须可追溯、可解释、可回滚。
  - 任何可能影响 `main` / `develop` 分支的操作，必须先向用户报告并等待确认。
  - 当用户的口语指令与本规范冲突时，AI 必须先指出冲突、说明风险，并等待用户明确确认后才能继续。
  - AI 不得替用户做出架构决策、发布决策、权限变更等高风险操作。
  - AI 生成的每一行代码都必须有明确的意图，禁止生成"看起来差不多"的代码。

  ### 1.3 优先级规则

  当多条规则同时适用时，按以下优先级裁决：

  1. **安全与数据保护**（最高）— 涉及敏感信息、数据丢失风险的规则
  2. **分支保护** — `main`/`develop` 分支的完整性
  3. **代码质量** — 编译通过、测试覆盖、无引入性 Bug
  4. **流程规范** — Commit 格式、PR 规范、命名约定
  5. **效率优化**（最低）— 性能优化、代码风格改进

  ========================================================================
  第二章：场景化行为约束
  ========================================================================

  ### 2.1 分支操作约束（强制）

  - **禁止** 直接向 `main` 或 `develop` 推送代码，所有改动必须通过 PR。
  - 新建功能分支必须从 `develop` 拉取，命名格式：`feature/<功能名>`。
  - 修复分支命名格式：`bugfix/<问题描述>`。
  - 重构分支命名格式：`refactor/<模块名>`。
  - 热修复分支必须从 `main` 拉取，命名格式：`hotfix/<问题描述>`，合并时先合 `main`，再合 `develop`。
  - **禁止** 对 `main`/`develop`/`release`/`hotfix` 执行 `git push --force`。
  - 在任何 git 操作前，先执行 `git branch --show-current` 确认当前分支。
  - 创建新分支前必须先执行 `git pull` 同步目标分支最新代码。

  ### 2.2 Commit 约束（强制）

  - 所有 commit message 必须遵循 Conventional Commits 格式：
    <type>(<scope>): <描述> 【<邮箱后四位>】
  - type 枚举值：`feat` | `fix` | `refactor` | `docs` | `test` | `style` | `chore`
  - scope 为可选模块名（如 `canvas`、`editor`、`core`、`docs`）
  - 描述使用中文或英文均可，简洁明了
  - **禁止** 生成类似 "update files"、"fix bug"、"changes" 等无意义 commit message
  - 一次 Commit 只做一件事，禁止将不相关的修改混在同一个 Commit 中

  ### 2.3 PR 创建约束（强制）

  - 功能分支完成后必须通过 `gh pr create` 或引导用户在 GitHub 上创建 PR。
  - PR 目标分支：日常功能 → `develop`；热修复 → `main`。
  - PR 标题必须使用 Conventional Commits 格式。
  - PR body 必须包含：变更说明、自测清单、关联任务（如有）。
  - 创建 PR 前必须确保本地分支已同步目标分支最新代码且无冲突。

  ### 2.4 冲突处理约束（强制）

  - 遇到 merge conflict 时，AI 必须：
    1. 先 `git fetch upstream develop` 获取最新代码
    2. 向用户报告冲突文件列表及冲突原因
    3. 逐个文件分析冲突内容，提供每个冲突的解决建议
    4. **禁止** 自动选择一方覆盖，必须向用户解释两个版本的差异
    5. 解决后要求用户确认再提交

  ### 2.5 代码生成约束

  - 生成代码前，必须先阅读相关模块的现有代码，理解架构风格和命名约定。
  - 不得引入新的第三方依赖，除非用户明确要求并确认。
  - 生成的代码必须通过 `flutter analyze` 无 error。
  - 涉及数据模型变更时，必须提供 Hive 迁移方案。

  ### 2.6 安全约束（强制）

  - **禁止** 在代码、commit、PR、Issue 中写入密钥、Token、密码、API Key 等敏感信息。
  - 如发现仓库中存在敏感信息，必须立即警告用户，并建议：
    1. 使用 `git filter-branch` 或 `BFG Repo-Cleaner` 清除历史记录
    2. 立即轮换已泄露的凭证
    3. 通知所有协作者重新 clone

  ### 2.7 版本号约束

  - 版本号遵循项目根目录 `VERSIONING.md` 中的语义化版本规范。
  - AI 不得自行修改 `pubspec.yaml` 中的版本号，除非用户明确指示。
  - 修改版本号时必须同步更新 `CHANGELOG.md`。

  ========================================================================
  第三章：人机协作流程
  ========================================================================

  ### 3.1 确认机制

  AI 执行以下操作前必须获得用户明确确认（文字回复"确认"/"好的"/"执行"等）：

  | 操作类别                   | 具体操作                                       | 确认级别 |
  |---------------------------|------------------------------------------------|----------|
  | 高风险操作                 | 推送代码、合并分支、删除分支、修改版本号       | 必须确认 |
  | 中风险操作                 | 创建 PR、修改配置文件、引入新依赖、重构核心模块 | 必须确认 |
  | 低风险操作                 | 读取文件、搜索代码、生成测试用例、格式化代码   | 可跳过   |
  | 信息查询                   | 查看分支状态、查看 commit 历史、查看 PR 列表    | 无需确认 |

  ### 3.2 沟通规范

  - AI 的回复必须使用与用户相同的语言（中文/英文）。
  - 报告问题时，必须同时提供：问题描述、影响范围、建议的解决方案。
  - 提出修改建议时，必须说明修改理由，而非仅给出修改内容。
  - 遇到不确定的情况，主动询问用户而非自行猜测。

  ### 3.3 上下文管理

  - 每次接手任务时，AI 应先确认当前分支和最新代码状态。
  - 操作涉及多个文件时，应在操作前列出完整的文件变更清单。
  - 长时间任务（超过 5 个步骤）应分阶段执行，每阶段完成后向用户汇报进度。

  ### 3.4 错误处理

  - AI 操作失败时，必须：
    1. 立即停止后续操作
    2. 报告错误原因和完整错误信息
    3. 提供回滚方案或修复建议
    4. 等待用户指示后再继续
  - 禁止在错误未解决的情况下继续执行后续步骤。

  ========================================================================
  第四章：多 Agent 协作规范
  ========================================================================

  ### 4.1 角色分配

  当多个 AI Agent 同时在本仓库工作时：

  | Agent 类型    | 职责范围                     | 允许操作的区域     |
  |---------------|------------------------------|---------------------|
  | 主 Agent      | 任务调度、整体协调           | 全仓库             |
  | 子 Agent      | 单一模块/文件的独立任务     | 仅被分配的模块     |
  | 审查 Agent    | 代码审查、安全检查           | 只读 + 评论        |

  ### 4.2 任务委派

  - 主 Agent 负责将任务拆解并分配给子 Agent，确保任务之间无依赖冲突。
  - 子 Agent 不得越权操作非分配区域。
  - 子 Agent 完成任务后必须将结果汇报给主 Agent，由主 Agent 统一整合。
  - 审查 Agent 只能通过 PR 评论提供反馈，不得直接修改代码。

  ### 4.3 冲突仲裁

  - 当多个 Agent 的操作产生冲突时，以主 Agent 的决策为准。
  - 如果是同级 Agent 之间的冲突（如两个子 Agent 修改同一文件），必须：
    1. 双方暂停操作
    2. 由主 Agent 评估冲突影响
    3. 主 Agent 指定一方优先或合并双方的修改
    4. 禁止子 Agent 自行覆盖其他 Agent 的修改

  ### 4.4 通信协议

  - Agent 之间通过共享工作目录进行通信，不使用外部通道。
  - 每个任务必须输出结构化的结果报告（修改了哪些文件、是否通过测试、是否存在风险）。
  - 跨 Agent 的依赖关系必须在任务开始前明确声明。

## 快速开始

```bash
# 1. Fork 并克隆仓库
git clone git@github.com:<你的用户名>/FadeMemo.git
cd FadeMemo

# 2. 添加上游仓库
git remote add upstream git@github.com:Stone-People-Like/FadeMemo.git

# 3. 安装依赖
flutter pub get

# 4. 运行项目
flutter run
```

## 分支模型

```
main（线上稳定版）
  ↑
  │ 合并
  │
develop（开发主干）
  ↑
  │ 合并
  │
feature/* / bugfix/* / refactor/*（功能分支）
```

| 分支 | 用途 | 保护级别 |
|------|------|----------|
| `main` | 线上稳定版，仅从 `release` / `hotfix` 合并 | 最高保护 |
| `develop` | 开发主干，日常功能分支的目标 | 强保护 |
| `feature/<功能名>` | 新功能开发 | 开放权限 |
| `bugfix/<问题描述>` | Bug 修复 | 开放权限 |
| `refactor/<模块名>` | 代码重构 | 开放权限 |
| `release/vX.Y.Z` | 版本发布 | 受限权限 |
| `hotfix/<问题描述>` | 线上紧急修复 | 受限权限 |

## 开发流程

### 日常功能开发

```
1. 从最新 develop 拉取功能分支
   git checkout develop && git pull upstream develop
   git checkout -b feature/<功能名>

2. 本地开发、自测，频繁提交到远程个人分支
   git add . && git commit -m "feat(module): 描述"
   git push -u origin feature/<功能名>

3. 定时拉取 develop 最新代码，提前解决冲突
   git fetch upstream develop && git merge upstream/develop

4. 开发完成 → 提交 PR，目标分支选择 develop

5. 评审通过 + 检测通过 → 合并到 develop

6. 合并完成后删除本地 + 远程废弃分支
   git branch -d feature/<功能名>
   git push origin --delete feature/<功能名>
```

### Bug 修复

```
1. 从 develop 拉取 bugfix 分支
   git checkout develop && git pull upstream develop
   git checkout -b bugfix/<问题描述>

2. 修复、自测 → 提 PR 合入 develop

3. 合并后清理分支
```

### 线上紧急修复（热修复）

```
1. 从 main 拉取 hotfix 分支
   git checkout main && git pull upstream main
   git checkout -b hotfix/<问题描述>

2. 修复、测试

3. 提 PR 先合并到 main（上线）

4. 再合并到 develop（同步修复）

5. 清理 hotfix 分支
```

### 版本发布

```
1. 从 develop 拉出 release 分支
   git checkout develop && git pull upstream develop
   git checkout -b release/vX.Y.Z

2. 全员回归测试，仅在 release 分支修复 Bug

3. 测试通过 → 合并 release 到 main 和 develop

4. 删除 release 分支
```

## Commit 规范

### 格式

```
<类型>(<模块>): <描述内容> 【<邮箱号前四位>】
```

### 类型

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 代码重构（不改变行为） |
| `docs` | 文档更新 |
| `test` | 测试相关 |
| `style` | 代码格式调整 |
| `chore` | 构建/工具/依赖变动 |

### 示例

```
feat(canvas): 新增错乱态 RGB 偏移动效 【1924】
fix(editor): 修复自动保存防抖失效问题 【1924】
docs(readme): 更新快速开始指南 【1924】
refactor(core): 重构遗忘引擎计算逻辑 【1924】
test(forgetting): 补充边界值测试用例 【1924】
```

## PR 规范

### 提交要求

- 标题使用 Conventional Commits 格式
- 目标分支选择正确（功能 → `develop`，修复 → `develop`/`main`）
- 已从目标分支拉取最新代码并解决冲突
- 自测通过，不影响已有功能
- Commit 历史整洁，避免无意义的提交

### 评审要求

| 目标分支 | 最低评审人数 |
|----------|-------------|
| `main` | 1~2 人 |
| `develop` | 至少 1 人 |

### 自动化检测

仓库已配置分支保护规则，PR 合并需要满足：
- 至少 1 人评审通过
- CI 检测通过（如已配置）
- 所有对话（评论）已解决

## 任务与分支绑定

1. 一个需求 / Bug 对应**唯一一条分支**，不多人共用一条长期分支
2. 一人可拥有多个分支，但一个分支只归属**一名主要负责人**

---

## 冲突处理

1. 代码冲突由**当前分支开发者**优先解决
2. 跨人 / 跨模块冲突，双方沟通后再修改，禁止私自覆盖他人代码
3. 公共组件、工具类修改，必须通知所有相关人员并加强评审

---

## 日常同步

- **每天开发前**：拉取 `develop` 最新代码，保持本地同步
- 不允许本地堆积大量代码，长时间不同步主干

---

## 分支清理

- 功能 / Bug 合并完成后 **24 小时内** 删除远程废弃分支
- 每周固定一次分支盘点，清理长期未使用的僵尸分支

---

## 禁止事项

- 禁止向 `main` / `develop` 直接 push（必须走 PR）
- 禁止强制推送（force push）到受保护分支
- 禁止长期多个分支互相嵌套合并
- 禁止跨人私自覆盖代码
- 禁止在 Commit 中包含敏感信息（密钥、Token 等）
