## 版本号格式

版本号采用三段式 `MAJOR.MINOR.PATCH`，并支持可选的预发布标识和构建元数据：

```
MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]
```

| 字段 | 说明 | 示例 |
|------|------|------|
| `MAJOR` | 主版本号 | `1`、`2`、`3` |
| `MINOR` | 次版本号 | `0`、`1`、`2` |
| `PATCH` | 修订号 | `0`、`1`、`2` |
| `PRERELEASE` | 预发布标识（可选） | `alpha.1`、`beta.3`、`rc.2` |
| `BUILD` | 构建号（可选） | `1`、`build.20260722` |

### 完整示例

```
1.0.0-alpha.1+1           # 首个 Alpha 预览版，构建号 1
1.0.0-beta.3+5            # 第三个 Beta 版，构建号 5
1.0.0-rc.2+8              # 第二个候选发布版
1.0.0+12                  # 正式版 1.0.0，构建号 12
1.1.0-beta.1+15           # 1.1 功能的 Beta 预览
2.0.0                     # 正式版 2.0.0（主版本升级）
```

## 版本号变更规则

### 1. MAJOR（主版本号）

**何时递增：**

- 引入不兼容的 API 变更
- 数据结构发生重大变更且不向后兼容
- 移除已有功能
- 核心算法或引擎重写
- 产品定位发生重大调整

**规则：** 递增 MAJOR 时，MINOR 和 PATCH 必须归零。

```
0.5.3  →  1.0.0   (首个正式版)
1.4.2  →  2.0.0   (不兼容的 API 变更)
```

### 2. MINOR（次版本号）

**何时递增：**

- 新增向后兼容的功能
- 新增平台支持（如新增桌面端、Web 端）
- 功能模块新增（如新增搜索、分类、导出）
- 重大 UI 改版
- 新增重要的可配置项

**规则：** 递增 MINOR 时，PATCH 必须归零。

```
1.0.3  →  1.1.0   (新增搜索功能)
1.2.0  →  1.3.0   (新增数据导出功能)
```

### 3. PATCH（修订号）

**何时递增：**

- Bug 修复
- 性能优化
- 安全修复
- 文档更新（不影响代码行为）
- 代码重构（不影响对外行为）

**规则：** 仅递增 PATCH。

```
1.1.0  →  1.1.1   (修复某个崩溃 Bug)
1.1.1  →  1.1.2   (修复另一个 Bug)
```

### 4. 特殊规则

| 场景 | 处理规则 |
|------|----------|
| **首个稳定版** | 从 `1.0.0` 开始，之前的 `0.x.x` 为开发预览期 |
| **紧急热修** | 可跳过预发布阶段直接发 PATCH，但需记录原因 |
| **多平台构建** | 各平台版本号保持一致，构建号可不同（Flutter `+build` 机制） |
| **回滚发布** | 不回退版本号，通过新的 PATCH 版本修复问题 |

## 预发布版本

预发布版本用于在正式发布前进行内部测试和外部预览。预发布版本的优先级低于对应的标准版本。

### 预发布标识优先级

```
alpha < beta < rc < 正式版
```

| 标识 | 名称 | 说明 |
|------|------|------|
| `alpha` | 内部预览版 | 功能不完整，仅供内部测试 |
| `beta` | 公开测试版 | 功能基本完整，允许外部用户参与测试 |
| `rc` | 候选发布版 | 功能冻结，仅修复阻塞性 Bug |

### 预发布版本递增规则

- 每次发布新的预发布版本时，递增预发布编号
- 从标准版本进入预发布阶段时，标准版本号保持不变

```
1.0.0-alpha.1  →  1.0.0-alpha.2  (Alpha 迭代)
1.0.0-alpha.3  →  1.0.0-beta.1   (进入 Beta 阶段)
1.0.0-beta.2   →  1.0.0-rc.1     (进入 RC 阶段)
1.0.0-rc.2     →  1.0.0          (正式发布)
```

### 开发期间的特殊标识

在 `0.x.x` 阶段（首个稳定版之前），预发布标识可简化为：

```
0.1.0-alpha.1
0.1.0-alpha.2
0.1.0                    (该功能里程碑完成)
0.2.0-alpha.1
```

## 构建元数据

构建元数据用于标识同一版本的不同构建产物，不参与版本优先级比较。

### Flutter 项目中的构建号

Flutter 使用 `+build` 语法：

```yaml
# pubspec.yaml
version: 1.0.0+1    # 版本号 1.0.0，构建号 1
```

| 平台 | 构建号映射 |
|------|-----------|
| Android | `versionCode`（整数，必须递增） |
| iOS / macOS | `CFBundleVersion` |
| Web | 无直接映射，仅用于标识 |
| Windows / Linux | 无直接映射，仅用于标识 |

### 构建号递增规则

- 每次构建（无论是否发布）递增构建号
- 构建号仅递增，不回退
- 构建号与版本号独立递增

```yaml
# 开发迭代
version: 0.0.1-alpha.2+1
version: 0.0.1-alpha.2+2
version: 0.0.1-alpha.2+3

# 发布新版本
version: 0.0.1-alpha.3+4
```

## Flutter 项目版本配置

### pubspec.yaml 示例

```yaml
name: fade_memo
description: 记忆遗忘模拟器 — 一款模拟人类记忆自然遗忘规律的跨端笔记 App

# 版本号格式: MAJOR.MINOR.PATCH+BUILD
version: 1.0.0-beta.3+12

environment:
  sdk: '>=3.2.0 <4.0.0'

dependencies:
  flutter:
    sdk: flutter
```

### 各平台自动映射

Flutter 构建时会自动将版本号映射到各平台配置：

| 平台 | 配置项 | 映射结果 |
|------|--------|----------|
| Android | `versionName` | `1.0.0-beta.3` |
| Android | `versionCode` | `12` |
| iOS | `CFBundleShortVersionString` | `1.0.0-beta.3` |
| iOS | `CFBundleVersion` | `12` |
| macOS | 同 iOS | 同 iOS |
| Windows | 文件版本 | `1.0.0.12` |

## 分支与版本对应关系

```
main  ──────────────────────────────────────────────►
       │           │           │           │
       ▼           ▼           ▼           ▼
    release/    release/    release/    release/
    1.0.0       1.1.0       1.1.1       2.0.0
       │           │           │           │
       ▼           ▼           ▼           ▼
    v1.0.0      v1.1.0      v1.1.1      v2.0.0
```

### 分支策略

| 分支 | 用途 | 版本号规则 |
|------|------|-----------|
| `main` | 主开发分支 | 始终保持最新稳定代码 |
| `release/x.y.z` | 发布分支 | 从 main 切出，版本冻结后仅修 Bug |
| `hotfix/x.y.z+1` | 热修分支 | 从对应 release 标签切出 |
| `feature/*` | 功能分支 | 开发期间不修改版本号 |

### 版本号变更流程

```
1. 功能开发完成 → 合并到 main（不修改版本号）
2. 准备发布 → 从 main 切出 release/x.y.z
3. 在 release 分支修改 pubspec.yaml 版本号
4. 发布前测试 → 发现 Bug → 在 release 分支修复
5. 测试通过 → 打 Git 标签 vX.Y.Z
6. 合并 release 分支回 main
7. 构建并发布
```

## Git 标签规范

### 标签格式

所有发布版本必须打 Git 标签，格式为 `v` + 版本号：

```bash
# 正式版
git tag v1.0.0
git tag v1.1.0
git tag v2.0.0

# 预发布版
git tag v1.0.0-alpha.1
git tag v1.0.0-beta.3
git tag v1.0.0-rc.2
```

### 附注标签（推荐）

使用附注标签（annotated tag）记录发布信息：

```bash
git tag -a v1.0.0 -m "Release v1.0.0 - 首个正式版

主要更新：
- 核心遗忘算法引擎
- 跨 5 平台支持
- 深浅双主题

构建号: +12"
```

### 标签推送

```bash
# 推送单个标签
git push origin v1.0.0

# 推送所有本地标签
git push origin --tags
```

## 发布流程

### 标准发布流程

```
┌─────────────────┐
│ 1. 功能开发完成  │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. 合并到 main  │
└────────┬────────┘
         ▼
┌──────────────────────────┐
│ 3. 更新 CHANGELOG.md      │
│    记录所有变更内容        │
└────────┬─────────────────┘
         ▼
┌──────────────────────────┐
│ 4. 更新 pubspec.yaml      │
│    版本号和构建号          │
└────────┬─────────────────┘
         ▼
┌──────────────────────────┐
│ 5. 打 Git 标签 vX.Y.Z    │
└────────┬─────────────────┘
         ▼
┌──────────────────────────┐
│ 6. 推送代码和标签         │
└────────┬─────────────────┘
         ▼
┌──────────────────────────┐
│ 7. CI/CD 自动构建各平台   │
└────────┬─────────────────┘
         ▼
┌──────────────────────────┐
│ 8. 发布到各平台应用商店   │
└──────────────────────────┘
```

### 预发布流程

```
1. 从 main 切出 release/x.y.z 分支
2. 修改版本号为 x.y.z-alpha.1 / beta.1 / rc.1
3. 构建并发布到测试渠道：
   - Android: Google Play Internal Testing / Firebase App Distribution
   - iOS: TestFlight
   - Web: 预览环境 / Vercel Preview
4. 收集反馈，修复 Bug
5. 递增预发布编号（alpha.2 / beta.2 等），重复步骤 3-4
6. 达到发布标准后，去掉预发布标识，进入正式版发布流程
```

版本号变更必须在 `CHANGELOG.md` 中有对应记录。格式参考 [Keep a Changelog](https://keepachangelog.com/zh-CN/1.0.0/)。

```markdown
## [1.1.0] - 2026-08-15

### 新增
- 移动端搜索功能
- 笔记分类标签

### 修复
- 桌面端窗口最小尺寸限制

## [1.0.1] - 2026-08-01

### 修复
- 修复长文本（>500字）渲染卡顿问题
- 修复深色主题切换时的闪烁

## [1.0.0] - 2026-07-22

### 新增
- 核心遗忘算法引擎
```
